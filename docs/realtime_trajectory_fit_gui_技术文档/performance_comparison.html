3.2 为什么 C++ 更快（5 点原理分析）

原因 1：消除 MATLAB 解释器开销

MATLAB 路径每姿态需要调用 inverseKinematics 多达 ~500 次（120+ 种子 × 4 权重阶段）。每次调用涉及：

  • 参数类型检查和验证（~0.1-0.5 ms）
  • mxArray ↔ 内部格式转换（~0.5-1 ms）
  • 函数分发和调用栈管理（~0.2 ms）

累计开销：500 calls × 258 poses × ~2 ms = ~258 秒解释器开销。C++ 中这些全部在编译时确定，运行时只有直接函数调用。

原因 2：KDL 解析雅可比 vs MATLAB 数值雅可比

MATLAB 的 inverseKinematics 内部使用数值雅可比（有限差分），每次 IK 迭代需要 6 次额外的 FK 计算。虽然我们的 C++ 实现也使用数值雅可比，但 KDL 的正运动学（O(N) 链式乘积）比 MATLAB 的 rigidBodyTree.getTransform 快得多：

  • MATLAB getTransform：遍历整个树结构，做大量字符串查找和矩阵乘法 → ~0.5-1 ms/次
  • KDL ChainFkSolverPos_recursive：预编译的链式 DH 参数乘积 → ~0.02-0.05 ms/次

一个 60 次迭代的 IK 求解需要 ~360 次 FK（6 雅可比列 × 60 迭代），MATLAB 需 ~180-360 ms，KDL 仅需 ~7-18 ms。

原因 3：FCL 原生碰撞检测 vs MATLAB 自定义几何

MATLAB 的 rtfg_collision 使用自定义几何启发式计算碰撞间隙（基于包围盒近似 + 距离公式），每次调用约 50-100 ms。FCL 使用高度优化的 BVH（包围体层次结构）树进行精确距离查询，配合 thread_local 对象缓存，单次调用仅需 ~15 ms。对于回放轨迹审计（~120 次采样），FCL 节省了 ~4-10 秒。

原因 4：无中间数据序列化

MATLAB 路径中，IK → 碰撞 → 代价计算 → 候选选择的每一步都需要在 MATLAB 工作空间和底层求解器之间传递数据（mxArray 分配/释放、类型转换）。C++ MEX 将所有计算保留在单一编译单元内：

MATLAB 路径数据流：
  IK → mxArray → MATLAB 工作空间 → mxArray → 碰撞检测 → mxArray → 代价计算
  （每次转换 ~0.5ms × 500次/姿态 × 258姿态 = ~64秒序列化开销）

C++ 路径数据流：
  IK → 碰撞 → 代价计算  （全部在 Eigen 矩阵 + C++ 栈/堆上，零拷贝）

原因 5：C++ 编译优化

  • -O3 优化：函数内联、循环展开、向量化（SIMD）
  • Eigen 表达式模板：矩阵运算在编译时融合为单一循环，消除临时变量分配
  • 栈分配：小矩阵（6×6、6×1）分配在栈上，无堆内存开销
  • 编译时常量：constexpr 权重数组、容限值在编译时求值
